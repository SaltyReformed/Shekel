# C-09 follow-ups: out-of-scope issues observed during implementation

**Origin:** Surfaced while implementing commit C-09 (TOTP replay prevention via
`auth.mfa_configs.last_totp_timestep`). Documented here for later, separate
follow-ups -- none of them were fixed in C-09 itself per the in-scope rule
(CLAUDE.md rule 6).

**Severity ordering:** Issue 1 (migration tooling) is the most likely to cause
real harm if left alone -- every future autogenerated migration carries a
landmine. Issues 2 and 3 are quieter background defects.

---

## Issue 1: `migrations/env.py` autogenerates a phantom `drop_table('alembic_version')`

### Symptom

Running `flask db migrate -m "<anything>"` produces an `upgrade()` that includes:

```python
def upgrade():
    op.drop_table('alembic_version')
    # ... real changes ...
```

and a matching `op.create_table('alembic_version', ...)` in `downgrade()`. If a
developer applies the migration without removing those lines, the next
`flask db current` and every subsequent migration command will fail because
Alembic has nowhere to record state. The C-09 migration was scaffolded by
`flask db migrate` and required a manual rewrite to strip the phantom drop.

### Root cause

`migrations/env.py` (lines 35-39, 42-53, 59-83):

```python
def include_name(name, type_, parent_names):
    """Include all schemas that Shekel uses in autogenerate."""
    if type_ == "schema":
        return name in ("ref", "auth", "budget", "salary", "system", None)
    return True
```

The `None` schema entry is the `public` schema. `version_table_schema="public"`
on lines 52 and 79 places the `alembic_version` table there. Autogenerate
compares the live `public` schema to `target_metadata`, finds
`alembic_version` is not declared in any model, and proposes dropping it.

The two interlocking decisions -- include `public` in autogenerate AND store
the version table in `public` -- collide. The fix is to keep both decisions
but explicitly exclude the version table from object-level comparison.

### Fix

Add an `include_object` filter to `env.py` and pass it into both
`context.configure(...)` blocks:

```python
def include_object(object_, name, type_, reflected, compare_to):
    """Skip Alembic's own bookkeeping table during autogenerate.

    The version table lives in ``public`` and is intentionally not
    declared in any model.  Without this filter, autogenerate sees
    a live table that ``target_metadata`` does not know about and
    proposes ``op.drop_table('alembic_version')`` -- which would
    brick migration tracking on the next ``flask db upgrade``.
    """
    if type_ == "table" and name == "alembic_version":
        return False
    return True
```

Then in `run_migrations_offline()`:

```python
context.configure(
    url=url,
    target_metadata=target_metadata,
    literal_binds=True,
    dialect_opts={"paramstyle": "named"},
    include_schemas=True,
    include_name=include_name,
    include_object=include_object,            # NEW
    version_table_schema="public",
)
```

And in `run_migrations_online()`:

```python
context.configure(
    connection=connection,
    target_metadata=target_metadata,
    process_revision_directives=process_revision_directives,
    include_schemas=True,
    include_name=include_name,
    include_object=include_object,            # NEW
    version_table_schema="public",
)
```

### Verification

After the fix, run `flask db migrate -m "<empty test>"`. The
`process_revision_directives` callback should detect the empty diff and skip
file generation (`"No changes detected -- skipping autogenerate."`). If a
file IS generated, the autogenerate filter is broken.

Also worth running `flask db migrate` after a no-op model change (e.g. a
trivial column reorder) and confirming the resulting migration contains ONLY
the intended diff -- no `drop_table('alembic_version')` line.

### Risk if not fixed

**High over time, zero today.** Every future autogenerated migration starts
with a destructive line. The C-09 migration caught it because I read the
generated file before applying. A future contributor who runs `flask db
migrate && flask db upgrade` without reviewing -- which is the documented
flow in `docs/runbook.md` -- will brick migrations on their dev DB and
potentially on production if the file ships unreviewed.

### Cross-refs

- `migrations/env.py` (current state)
- Documentation of expected flow: `CLAUDE.md` "Database migrations" section
- Coding standard: `docs/coding-standards.md` "Review auto-generated
  migrations. Verify intended changes, no phantom diffs..."

## Issue 2: `transaction_entries.created_at` and `updated_at` are model-vs-DB drift

### Symptom

`flask db migrate` after no model changes still produces:

```python
op.alter_column('transaction_entries', 'created_at',
                existing_type=postgresql.TIMESTAMP(timezone=True),
                nullable=True,
                existing_server_default=sa.text('now()'),
                schema='budget')
op.alter_column('transaction_entries', 'updated_at',
                existing_type=postgresql.TIMESTAMP(timezone=True),
                nullable=True,
                existing_server_default=sa.text('now()'),
                schema='budget')
```

These were stripped from the C-09 migration because they're not C-09 work.

### Root cause

Model declares the columns without `nullable=False`. The DB has them as
NOT NULL (the migration that created the table got it right; the model
declaration drifted later).

`app/models/transaction_entry.py:83-88`:

```python
created_at = db.Column(db.DateTime(timezone=True), server_default=db.func.now())
updated_at = db.Column(
    db.DateTime(timezone=True),
    server_default=db.func.now(),
    onupdate=db.func.now(),
)
```

Compare with the equivalent columns elsewhere in the codebase, e.g.
`app/models/user.py:24-29`:

```python
created_at = db.Column(db.DateTime(timezone=True), server_default=db.func.now())
updated_at = db.Column(
    db.DateTime(timezone=True),
    server_default=db.func.now(),
    onupdate=db.func.now(),
)
```

User has the same model declaration but the DB columns there are also
nullable -- consistent with the model. So the inconsistency is local to
`transaction_entries`: model says nullable, DB says NOT NULL, and they
disagree.

In other words, **two independent issues** are tangled here:

a. The DB has `transaction_entries.created_at`/`updated_at` NOT NULL while
   the model says nullable -- the model is wrong, the DB is right.
b. Most other tables (users, accounts, transactions, etc.) have
   `created_at`/`updated_at` with the same model declaration but
   nullable in the DB -- the model and DB agree, but both are weaker
   than they should be. NOT NULL is the right invariant for an
   audit-trail timestamp, but adding NOT NULL is a destructive
   migration that needs explicit approval per `docs/coding-standards.md`.

This follow-up addresses (a) only. (b) is a separate question of "should
every audit timestamp be NOT NULL across the codebase" that should be
resolved as a project-wide standard, not as a side effect of fixing this
drift.

### Fix

Update the model to match the DB. **No DB migration needed** -- the DB is
already correct.

`app/models/transaction_entry.py:83-88`:

```python
created_at = db.Column(
    db.DateTime(timezone=True), nullable=False, server_default=db.func.now(),
)
updated_at = db.Column(
    db.DateTime(timezone=True),
    nullable=False,
    server_default=db.func.now(),
    onupdate=db.func.now(),
)
```

### Verification

After the fix:

1. Run `flask db migrate -m "<empty test>"`. The autogenerate should detect
   no differences and `process_revision_directives` should skip file
   generation. If a file IS generated and contains an `alter_column`
   targeting `transaction_entries`, model and DB are still mismatched.

2. Run `pytest tests/test_models/ tests/test_routes/` -- any test that
   inserts a `TransactionEntry` without setting timestamps should still
   pass because `server_default=db.func.now()` populates them at the DB
   level on INSERT.

3. Inspect `\d budget.transaction_entries` in psql. Expect:

   ```
    created_at | timestamp with time zone | not null | now()
    updated_at | timestamp with time zone | not null | now()
   ```

### Risk if not fixed

**Medium.** A future autogenerated migration could accidentally widen
`transaction_entries.created_at`/`updated_at` to nullable if the developer
applies the autogen output without scrutiny. Once nullable, NULL rows
become possible, and the audit-trail invariant ("every entry has a
creation timestamp") silently weakens. There is no test that asserts NOT
NULL on these columns, so the regression would be invisible until someone
queries the audit log and finds NULL timestamps.

### Cross-refs

- `app/models/transaction_entry.py:83-88` (the drift)
- `docs/coding-standards.md` "NOT NULL by default. Every new column should
  be NOT NULL unless there is a specific reason for nullability."
- Companion question (b) above is broader and worth its own discussion.

## Issue 3: Pre-existing `W9016 missing-type-doc` warnings in `mfa_service.py`

### Symptom

`pylint app/services/mfa_service.py --fail-on=E,F` reports seven warnings:

```
app/services/mfa_service.py:164:0: W9016: "plaintext_secret" missing in parameter type documentation (missing-type-doc)
app/services/mfa_service.py:176:0: W9016: "encrypted_secret" missing in parameter type documentation (missing-type-doc)
app/services/mfa_service.py:188:0: W9016: "email, issuer, secret" missing in parameter type documentation (missing-type-doc)
app/services/mfa_service.py:202:0: W9016: "uri" missing in parameter type documentation (missing-type-doc)
app/services/mfa_service.py:351:0: W9016: "count" missing in parameter type documentation (missing-type-doc)
app/services/mfa_service.py:379:0: W9016: "codes, rounds" missing in parameter type documentation (missing-type-doc)
app/services/mfa_service.py:398:0: W9016: "code, hashed_codes" missing in parameter type documentation (missing-type-doc)
```

These are all pre-existing -- they predate C-09. The C-09 verifier additions
(`verify_totp_code`, `verify_totp_setup_code`, the helper, the enum) all
include `(type)` annotations and do not contribute to this list.

### Root cause

The seven flagged docstrings describe parameters in prose but do not include
explicit type annotations. Pylint's `napoleon`-style doc check expects either:

- A function-level type annotation in the signature, OR
- A `(type)` token after the parameter name in the docstring.

The flagged functions do neither. Example: `encrypt_secret`, currently:

```python
def encrypt_secret(plaintext_secret):
    """Encrypt a TOTP secret for database storage.

    Args:
        plaintext_secret: The base32-encoded secret string to encrypt.

    Returns:
        bytes: The Fernet-encrypted ciphertext.
    """
    return get_encryption_key().encrypt(plaintext_secret.encode("utf-8"))
```

The Returns line includes `bytes:` so pylint doesn't flag the return type.
The Args line has no `(str):` after `plaintext_secret`, hence the warning.

### Fix

Add `(type)` annotations to each Args entry in the seven functions. This is
a mechanical, behavior-free change. Sample for `encrypt_secret`:

```python
def encrypt_secret(plaintext_secret):
    """Encrypt a TOTP secret for database storage.

    Args:
        plaintext_secret (str): The base32-encoded secret string to encrypt.

    Returns:
        bytes: The Fernet-encrypted ciphertext.
    """
    return get_encryption_key().encrypt(plaintext_secret.encode("utf-8"))
```

Apply the same pattern to:

| Function | Parameters needing types |
|---|---|
| `encrypt_secret` (line 164) | `plaintext_secret (str)` |
| `decrypt_secret` (line 176) | `encrypted_secret (bytes)` |
| `get_totp_uri` (line 188) | `secret (str)`, `email (str)`, `issuer (str)` |
| `generate_qr_code_data_uri` (line 202) | `uri (str)` |
| `generate_backup_codes` (line 351) | `count (int)` |
| `hash_backup_codes` (line 379) | `codes (list[str])`, `rounds (int \| None)` |
| `verify_backup_code` (line 398) | `code (str)`, `hashed_codes (list[str])` |

### Verification

After the fix:

```bash
pylint app/services/mfa_service.py --fail-on=E,F
```

Expect zero `W9016` warnings on `mfa_service.py`. The pylint score for the
file should rise from 9.80 to ~10.00.

### Risk if not fixed

**Low.** It's a documentation hygiene issue, not a behavioral one. The
project's coding standard says "Do not decrease the current score";
addressing these warnings raises the baseline so future work has a
slightly higher bar to meet.

### Cross-refs

- `app/services/mfa_service.py` lines listed above
- `docs/coding-standards.md` "Pylint compliance is mandatory"
- `docs/coding-standards.md` "Fix Pylint violations, do not suppress them"
