## 0.2 Standards and roadmap

### Status workflow values

- **Projected** -- a transaction not yet reconciled; the default state for future transactions (`app/enums.py:23`).
- **Paid** (DONE) -- expense whose checking outflow has occurred; comment notes the rename from "done" (`app/enums.py:24`).
- **Received** -- income that has been deposited (`app/enums.py:25`).
- **Credit** -- expense paid via credit card; does not affect the checking balance (`app/enums.py:26`).
- **Cancelled** -- a projected transaction that will not occur; excluded from balance calculations (`app/enums.py:27`).
- **Settled** -- archived/fully reconciled; already reflected in the anchor balance (`app/enums.py:28`).

`CLAUDE.md:105-106` summarizes the workflow as `projected -> done|credit|cancelled` and `done|received -> settled`.

### Transfer invariants (verbatim from CLAUDE.md)

From `CLAUDE.md:130-138`:

```text
## Transfer Invariants

**CRITICAL INVARIANTS (violating any one is a critical bug):**

1. Every transfer has exactly two linked shadow transactions (one expense, one income).
2. Shadow transactions are never orphaned and never created without their sibling.
3. Shadow amounts, statuses, and periods always equal the parent transfer's.
4. No code path directly mutates a shadow. All mutations go through the transfer service.
5. Balance calculator queries ONLY budget.transactions. NEVER also query budget.transfers.
```

### Rules the standards documents impose on calculation code

- Use `Decimal`, never `float`, for all monetary amounts (`docs/coding-standards.md:13`).
- Construct Decimals from strings; `Decimal(0.1)` introduces float imprecision (`docs/coding-standards.md:14-15`).
- Type hints on all signatures with `Decimal` (not `float`) for monetary values (`docs/coding-standards.md:16-19`).
- Do not rely on truthiness for business logic; `0` and `None` differ (`docs/coding-standards.md:20-22`).
- No magic numbers or strings; named constants for every business-rule value (`docs/coding-standards.md:26-30`).
- Keep functions focused; over 50 lines requires decomposition review, over 100 needs justification (`docs/coding-standards.md:30-32`).
- Prefer guard clauses over deep nesting; max nesting depth 3 in business logic (`docs/coding-standards.md:33-34`).
- Substantive docstrings; financial calculations document the formula or business rule (`docs/coding-standards.md:43-45`).
- Comments explain why, not what (`docs/coding-standards.md:46-47`).
- Catch specific exceptions; never `except Exception:` (`docs/coding-standards.md:52-53`).
- All queries must be user-scoped; missing ownership checks are IDOR vulnerabilities (`docs/coding-standards.md:79-80`).
- Always filter soft-deleted records (`docs/coding-standards.md:81-82`).
- Use SQLAlchemy ORM; no raw SQL strings in application code (`docs/coding-standards.md:83`).
- Prevent N+1 queries via `joinedload`/`subqueryload`/`selectinload` (`docs/coding-standards.md:84-86`).
- NUMERIC(12,2) for all monetary columns; never Float, Integer, or bare Numeric (`docs/coding-standards.md:92`).
- CHECK constraints on every financial column; schema must match Marshmallow ranges (`docs/coding-standards.md:93-95`).
- Marshmallow schema for every state-changing route before any database operation (`docs/coding-standards.md:106-108`).
- Validate FK existence before commit; verify ownership (`docs/coding-standards.md:109-110`).
- Range validation must match between schema and database (`docs/coding-standards.md:111-112`).
- IDs for logic, strings for display only; never compare ref-table `name` columns (`docs/coding-standards.md:153-155`).
- Templates are for display, not computation; never perform financial calculations in Jinja (`docs/coding-standards.md:163-164`).
- No lazy-loaded queries in templates; eager-load before iteration (`docs/coding-standards.md:167-168`).
- Monetary values in JS are display-only; arithmetic happens server-side in Python (`docs/coding-standards.md:194-195`).
- Service tests assert computed values with exact expectations; include the arithmetic in a comment (`docs/testing-standards.md:60-65`).
- Edge case tests assert the specific edge behavior, not just non-crash (`docs/testing-standards.md:67-71`).

## 0.3 Behavioral expectations

The audit verifies each expectation below against the actual code in Phases 3 and 5. Failure of any of them is a finding regardless of whether an implementation plan documents it.

### Developer-stated

E-01: Transfer to a debt account splits two ways.

A transfer out of a checking account and into a mortgage (or any debt account) reduces the checking balance by the full payment amount, including principal, interest, escrow, and any other components. The destination debt account records the full amount as received but only the principal portion of the payment reduces the loan principal balance. Interest, escrow, and any other non-principal components are recorded as part of the transfer but do not reduce loan principal.

Source: @docs/financial_calculation_audit_plan.md:131-139 (developer-stated to auditor).

E-02: Monthly payment is stable inside the fixed-rate window of an ARM.

A 5/5 ARM during the first five years has one monthly payment value. The amortization engine, however many times it is called and from whichever entry point, produces the same payment for any month in that window. Fluctuation by even a few cents is a finding.

Source: @docs/financial_calculation_audit_plan.md:140-145 (developer-stated to auditor).

E-03: Loan principal updates as confirmed transfers occur.

When a transfer to a debt account is settled, the real loan principal reflects the principal portion of that payment. Whether the update happens by writing a stored column or by recomputing from confirmed payments is an implementation choice; the audit must determine which approach the code uses, whether it is consistent across display sites, and whether it is correct.

Source: @docs/financial_calculation_audit_plan.md:146-152 (developer-stated to auditor).

E-04: An account balance is the same number on every page that shows it.

The checking-account balance for the current pay period on the grid, on `/savings`, and on `/accounts` is the same number for the same inputs. If two views are intentionally different (one shows the floor, another shows the projection), the difference is labeled and documented; an unlabeled difference is a finding.

Source: @docs/financial_calculation_audit_plan.md:153-158 (developer-stated to auditor).

### From coding-standards

E-10: Monetary amounts must use `Decimal`, never `float`.

Source: @docs/coding-standards.md:13.

E-11: Decimal values for money must be constructed from strings; `Decimal(0.1)` introduces float imprecision and is forbidden, including in hardcoded values, test assertions, seed data, and defaults.

Source: @docs/coding-standards.md:14-15.

E-12: Financial logic must distinguish zero from None; a zero balance is a real balance and is not the same as a missing balance, so business logic must not rely on truthiness checks against monetary values.

Source: @docs/coding-standards.md:20-22.

E-13: Queries on tables with `is_deleted` must filter soft-deleted records (`Model.is_deleted.is_(False)`) unless deleted records are explicitly needed; this rule governs every aggregate that consumes those tables.

Source: @docs/coding-standards.md:81-82.

E-14: Monetary database columns must be declared `NUMERIC(12,2)`; `Float`, `Integer`, and bare `Numeric` are forbidden, so stored monetary precision is fixed at two decimal places by the schema.

Source: @docs/coding-standards.md:92.

E-15: Reference-table conditionals (status, transaction type, account type, and similar lookup tables) must compare integer IDs, never string `name` columns; boolean ref columns and FK references are the supported shape for category logic.

Source: @docs/coding-standards.md:153-155.

E-16: Financial calculations must execute in routes or services in Python with `Decimal`; Jinja templates display computed values but never perform monetary arithmetic themselves.

Source: @docs/coding-standards.md:163-164.

E-17: Financial arithmetic happens server-side in Python; JavaScript treats monetary values as display-only and never computes them.

Source: @docs/coding-standards.md:196-197.

## 0.5 Cross-plan contradictions and open plan questions

Each entry pairs two or more watchlist claims that touch the same code location or concept but state different rules; cross-plan questions are also in `09_open_questions.md` (Q-02 onward).

C-01: Carry-forward envelope semantics: data-layer settle vs display-layer envelope view
- Entries in tension: W-095 / W-096 / W-101 (in `carry_fwd_design`), W-118 (in `carry_fwd_impl`), W-127 / W-128 / W-130 / W-131 / W-145 (in `envelope_view`).
- Disagreement: `carry_fwd_design` Option F (and `carry_fwd_impl`'s implementation of it) settles the source row to status DONE/RECEIVED with `actual_amount = entries_sum`, leaves `pay_period_id` unchanged, and bumps the target canonical's estimate; `envelope_view` keeps the post-33cd21e behavior where the source moves whole into the target period and the cell aggregates canonical + carried members for display, with a new `carried_from_period_id` column. Both architectural shapes cannot coexist for the same envelope template.
- Adjudication: resolved (2026-05-13) -- `carry_fwd_impl` (Option F) is current. `envelope_view` is superseded: its display goal was reached by Option F's bumped-canonical mechanism and its data-model additions (`carried_from_period_id`, `EnvelopeCell`, `grid_aggregation`) were never built. See A-02 in `09_open_questions.md`.

C-02: Recurrence skip rule for `is_override` rows
- Entries in tension: W-120 (in `carry_fwd_impl`), W-133 / W-134 / W-147 (in `envelope_view`).
- Disagreement: `carry_fwd_impl` leans on the existing skip-on-override rule (`is_override=True` blocks regeneration regardless of provenance); `envelope_view` narrows the rule so carried-only overrides do NOT block canonical generation, only non-carried overrides (manual edits) do.
- Adjudication: resolved (2026-05-13) -- skip rule unchanged; any `is_override=True` row continues to block regeneration. `envelope_view`'s narrowing was never implemented because its data model never landed (see C-01); under Option F a single override flag is sufficient because the target canonical IS the canonical, not a separate carried sibling. See A-03 in `09_open_questions.md`.

C-03: ARM `current_principal` source for projection and debt strategy
- Entries in tension: W-222 (in `section5`), W-060 (in `arm_anchor`).
- Disagreement: `section5` says current principal must be derived from confirmed payments via engine replay and used in place of the static `LoanParams.current_principal` for projections; `arm_anchor` says for ARM loans `_compute_real_principal()` must return `current_principal` directly, skipping the payment replay because forward-from-origination is mathematically wrong without rate history.
- Adjudication: resolved (2026-05-13) -- both rules apply by loan type. ARM loans use the stored `current_principal` directly (per `arm_anchor` 3F); fixed-rate loans walk the schedule from origination using confirmed `PaymentRecord`s (per `section5`). This dual policy is documented at `app/services/amortization_engine.py:848-861`. See A-04 in `09_open_questions.md`.

C-04: ARM monthly payment computation method
- Entries in tension: W-218 / W-219 / W-220 / W-234 (in `section5`), W-042 / W-043 / W-044 / W-047 / W-048 (in `arm_anchor`).
- Disagreement: `section5` describes the engine as replaying actual payments from origination through to today and re-amortizing at every rate change; `arm_anchor` introduces an anchor-reset at today (using `current_principal` and `current_rate`) and recomputes monthly payment from there forward, treating pre-anchor rows as approximate. These produce different monthly payment values inside an ARM's fixed-rate window when rate history is incomplete.
- Adjudication: resolved (2026-05-13) -- `arm_anchor`'s anchor-reset is current at all eight ARM monthly-payment call sites (`amortization_engine.py:440, :491, :512, :697, :952`; `balance_calculator.py:225`; `loan_payment_service.py:251`; `routes/loan.py:1225`). See A-05 in `09_open_questions.md`.
